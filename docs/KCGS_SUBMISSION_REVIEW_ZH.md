## 1. 结论先行

如果以当前材料直接投稿，最接近学会审稿分级中的 **“전면수정 후 재투고（全面修改后重新投稿）”**，
而不是“修改后即可录用”。项目已经达到可运行、可复验的研究原型水平，但尚未达到可以稳健支持主要因果
结论的论文水平。

当前最合适的论文定位是：

> 在受控 Minecraft 场景中，对“LLM 候选任务—服务器契约—确定性执行—运行时恢复”的可复现机制验证。

当前证据不支持以下更强表述：

- IBC 在任意开放世界任务中普遍优于规则系统；
- IBC 已经提高玩家沉浸感、信任或社会临场感；
- 100% 安全率代表系统不存在区域外修改、错误物品转移或实体伤害；
- 162 个主实验 episode 等价于 162 个独立样本；
- v0.4 是经过外部时间戳证明的前瞻性预注册实验。

建议题目：

- 韩文：`오픈월드 게임에서 검증 가능한 LLM 기반 NPC 실행을 위한 명령–행동 계약`
- 英文：`Instruction–Behavior Contracts for Verifiable LLM-Driven NPC Execution in Open-World Games`
- 中文工作题目：`面向开放世界游戏中可验证 LLM-NPC 执行的指令—行为契约`

## 3. 当前项目的强项

### 3.1 工程实现已经形成完整闭环

- 服务器权威的任务白名单、动态前置条件和动作边界；
- LLM 只提出高层候选任务，不直接获得逐 tick 世界控制权；

这种克制是项目的重要优点，应在韩文/英文稿中保留。

### 3.3 复验结果通过

本次审计重新执行并确认：

- Gradle 全量重跑：12 个测试套件、33 个测试，0 失败、0 跳过；
- Python 分析测试：5 个测试，全部通过；
- 使用 `artifacts/experiments/v0.4-with-a2/curated` 重建分析结果；
- 7 个 CSV 与 Rule-BT 复刻 JSONL 的 SHA-256 均与仓库版本一致；
- Markdown/JSON 汇总文件只因生成时间和当前 Git 提交号不同而产生哈希差异。

## 5. 必须在投稿前处理的关键问题

### P0-1：隔离系统自变量

主实验中 LLM-Schema 与 Maid-IBC 分别调用模型。对原始事件日志按“场景 × 重复号”配对后，
15/54 对候选任务并不完全相同，差异均表现为半径 12 与 24。当前受控布局使这些半径差异大多没有
改变结果，但它仍然破坏了“只切换 IBC 模块”的严格实验解释。

处理方式：模型候选只生成一次并冻结；所有架构变体回放完全相同的 `JobSpec`、场景、种子和扰动时点。

### P0-2：把行为收益与失败码差异分开

LLM-Schema/A2 相对完整系统的 6 个不匹配由两个场景组成：

1. `chest_removed_after_accept` 的 3 次重复只是 `TARGET_DISAPPEARED` 与预期 `TARGET_NOT_FOUND` 不同；
2. `recoverable_target_change` 的 3 次重复才是“失败”与“恢复后完成”的任务效用差异。

处理方式：主要终点使用世界效用和安全目标；失败码精确匹配只作为诊断指标。论文不能把全部 11.1 个
百分点都解释为任务完成能力提升。

### P0-3：扩展独立扰动场景

当前只有 3 个扰动场景，真正区分恢复效用的只有 1 个。三次重复能测稳定性，不能把独立样本数从 1
变成 3。

处理方式：至少冻结 12 个机制不同的扰动模板，并在多个布局种子上运行；统计推断以场景模板为聚类
单位。最终数量须在看新结果之前通过功效/精度分析确定。

### P0-4：重建独立安全判定器

现有 `safetySatisfied` 主要检查 21×8×21 测试区内的非白名单方块变化，没有完整观测：

- 区域外方块变化；
- 非目标物品或错误容器的转移；
- 对玩家、友方或其他实体的伤害；
- 锁箱、权限、维度和任务取消后的后续动作；
- 超过请求数量的合法方块破坏。

处理方式：建立与执行器分离的事件账本和不变量判定器，逐条输出违规证据，而不是只输出布尔值。

### P0-5：修复协议与实现不一致

冻结 v0.4 中，`job_spec.schema.json` 对候选任务只要求 `type`，而 Java 解码器对参数化任务还强制
`target/quantity/radius`；Schema 也没有表达 `dialogue_act` 与 `candidate_job` 的条件关系。因此
“严格等价校验”的表述不成立。

处理方式：保留 v0.4 原件；在 v0.5 使用条件 Schema，并增加“Schema 验证结果必须与 Java 解码器
一致”的属性/参数化测试。

### P0-6：建立可信的前瞻性冻结

v0.4 本地冻结时间早于主实验，但代码提交发生在结果之后，且快照记录工作树并非干净。它可以称为
“本地协议冻结”，不宜称为具有外部证据的“公开预注册”。

处理方式：在采集 v0.5 数据前完成干净 Git 提交、版本标签、公开时间戳登记和只读制品发布。

## 6. 代码级审查发现

以下问题不要求回写 v0.4，但必须在 v0.5 分支修复并新增测试：

| 优先级 | 位置 | 问题 | 风险 |
|---|---|---|---|
| P0 | `llm/JobSpecJsonCodec` 与 Schema | 两套校验规则不等价 | JVR 和“Schema-only”定义不清 |
| P0 | `contract/ContractCompiler` | 采集前只检查能否再放 1 个物品，不检查请求总量容量 | 契约可能先接受、后因可预见容量不足失败 |
| P1 | 三个 executor 的 `restore` | 重启后重新获得完整 90 秒预算 | “父契约总超时”可被重启延长 |
| P1 | `TaskContract.restored` | 恢复时丢失原前置条件、目标谓词和不变量 | 重启后的契约审计信息不完整 |
| P1 | 本地恢复计数 | `maxLocalRetries` 不是所有恢复的统一总上限 | 文档中的有限重试语义不精确 |
| P1 | `AiPartnerConfig.load` | 无效配置日志称禁用，但实际返回默认配置 | 有默认密钥环境变量时可能意外启用远端模型 |
| P1 | endpoint 校验 | 允许向非本地明文 HTTP 端点发送 Bearer 密钥 | 安全配置边界不足 |
| P1 | 测试目录 | 没有核心 executor、实体持久化和网关集成测试 | 当前 33 个测试不能覆盖主要世界状态机风险 |

## 10. 是否需要玩家实验

对于“可靠执行机制”的系统论文，玩家实验不是投稿的绝对前提。当前更稳妥的选择是删除沉浸感、信任和
社会临场感方面的效果主张，把玩家研究列为后续工作。

如果论文要声称改善玩家体验，则必须另开阶段：伦理审批/豁免、招募与功效分析、知情同意、量表、行为
指标、访谈编码和数据保护。不能用开发者试玩或界面截图替代玩家实证。
