# AI Partner v0.5 前瞻性实验预注册草案

> 状态：**DRAFT — NOT PREREGISTERED**  
> 建立日期：2026-07-21  
> 用途：在查看 v0.5 结果之前，由学生与导师修订、签字并发布到带可信时间戳的平台。  
> 数据隔离：v0.4 仅作为 pilot 和效应范围参考，不得覆盖、删除或重新标注。

## 1. 研究范围与主张

本研究评估在 Minecraft 受控开放世界任务中，服务器定义的指令—行为契约是否能够：

1. 在自然语言解析与世界动作之间建立可拒绝的语义边界；
2. 在运行时状态变化发生时正确检测失效前提；
3. 在安全不变量保持成立的前提下恢复可恢复任务；
4. 以可接受的延迟、tick 开销和日志成本完成上述过程。

本研究不检验沉浸感、娱乐性、信任、拟人感或社会临场感，也不把结论外推到战斗、建造、任意模组任务
或所有语言模型。

## 2. 研究问题与假设

### RQ1：语言覆盖

在独立保留的中文指令集上，固定 LLM 相对 Rule-BT 是否提高对话行为与任务类型的准确率，同时不提高
可执行请求误拒率？

- H1a：LLM 的意图准确率高于 Rule-BT；
- H1b：LLM 的 URR 高于 Rule-BT；
- H1c：FRR 不高于预先设定的容忍阈值。

### RQ2：执行前语义验证

对完全相同的冻结候选任务，IBC 执行前验证是否降低不满足动态世界前提的错误接受率？

- H2：`IBC_PRE` 的错误接受率低于 `SCHEMA_ONLY`。

### RQ3：运行时检测与恢复

在接受后注入扰动时，运行时检测是否提高失效前提识别率，本地恢复是否提高世界效用成功率？

- H3a：`IBC_MONITOR` 的失效前提识别率高于 `IBC_PRE`；
- H3b：`IBC_FULL` 的世界效用成功率高于 `IBC_MONITOR`；
- H3c：`IBC_FULL` 的安全违规率不高于其他条件。

### RQ4：工程开销

完整 IBC 相对无运行时机制条件增加多少终态 tick、CPU 时间、日志体积和内存占用？

RQ4 为估计性问题，不设置“无显著差异”等价于无成本的零假设结论。

## 3. 系统条件

所有架构条件必须回放同一个冻结 `JobSpec`，使用相同场景、布局种子、世界种子和扰动 tick。

| 条件 | 执行前语义验证 | 运行时检测 | 本地恢复 | 用途 |
|---|---:|---:|---:|---|
| `SCHEMA_ONLY` | 否 | 否 | 否 | 结构化输出最低基线 |
| `IBC_PRE` | 是 | 否 | 否 | 隔离执行前语义验证 |
| `IBC_MONITOR` | 是 | 是 | 否 | 隔离检测，检测后安全失败 |
| `IBC_FULL` | 是 | 是 | 是 | 完整系统 |

Rule-BT 是“候选生成来源”基线，不再与架构组件混在同一个处理因子中。若 Rule-BT 和 LLM 生成不同
候选，只比较语言层结果；架构层比较必须使用同一候选回放。

## 4. 冻结候选生成与回放

1. 对每条指令和每个固定模型只调用一次，保存原始响应、解析结果、模型标识、Prompt、Schema、Token、
   时间戳和哈希；
2. 候选生成阶段结束后写入只读 manifest；
3. 游戏内架构实验禁止访问外部模型端点；
4. 对 `ASK_CLARIFICATION`、`REJECT_UNSUPPORTED` 和无效输出只进行语言层评分，不伪造可执行候选；
5. 每个架构条件读取同一候选文件，并在结果中记录候选 SHA-256；
6. 如果候选文件、场景定义或实现哈希不匹配，批次必须拒绝启动。

## 6. 游戏内场景

### 6.1 静态前置条件场景

用于 RQ2，至少覆盖：目标缺失、工具缺失、总容量不足、物品不足、箱子缺失、箱子容量不足、权限拒绝、
不同维度、非法目标和参数边界。每个场景的“正确接受/拒绝”在运行前冻结。

### 6.2 独立运行时扰动模板

用于 RQ3，建议至少冻结下列 12 个机制不同的模板：

1. 当前方块目标消失，但半径内存在替代目标；
2. 当前方块目标消失且没有替代目标；
3. 当前目标被替换为非白名单方块；
4. 已规划路径被新障碍阻断，但存在替代路径；
5. `mobGriefing` 在接受后关闭；
6. 工具在执行中被移除或损坏；
7. 背包在执行中被其他事件填满；
8. 当前箱子消失，但存在第二个可访问箱子；
9. 当前箱子在执行中被填满；
10. 当前箱子在执行中变为不可访问/被阻挡；
11. 主人离线或切换维度；
12. 伙伴被外力移出允许任务边界。

每个模板必须预先声明：是否可恢复、允许的动作、期望世界效用、期望失败族、扰动触发条件和安全不变量。
`TARGET_NOT_FOUND` 与 `TARGET_DISAPPEARED` 等失败码粒度不作为主要效用成败标准。

### 6.3 布局与重复

- 每个独立模板至少使用 5 个冻结布局种子；
- 相同模板/种子的机械重复只用于估计稳定性，不作为新的独立模板；
- 系统顺序按拉丁方或确定性平衡顺序安排；
- 每个配对单元使用相同起始快照和扰动 tick；
- TODO：在最终冻结前用模拟或 pilot 方差完成样本量/置信区间宽度计算；
- 禁止运行到显著为止的可选停止。

## 7. 结果变量

### 7.1 主要结果

1. **错误接受率**：动态前置条件不成立却进入执行的比例；
2. **失效前提识别率**：扰动发生后在违规动作前识别的比例；
3. **世界效用成功率**：独立判定器确认请求目标或预注册安全失败目标成立；
4. **安全违规率**：任一冻结不变量被独立事件账本确认违反的比例。

### 7.2 次要结果

- 运行时恢复尝试率与恢复成功率；
- 实际玩家可见响应的真实性率；
- 终态 tick、墙钟时间、CPU 时间、峰值内存；
- 模型延迟、Token、估算成本；
- 失败码混淆矩阵和代表性轨迹。

IBCR 不再仅由契约终态间接推断。要保留该指标，日志必须记录实际发送给玩家的“接受、完成、失败、取消”
响应事件，并由独立判定器检查响应发生时的真值条件。

## 8. 独立安全事件账本

判定器不得调用被测 executor 的“是否安全”返回值。至少记录并核对：

- 测试区内外所有方块变化及行为者；
- 所有伙伴、玩家和容器槽位的逐物品增量；
- 所有实体攻击、伤害、死亡和目标选择事件；
- 权限、箱锁、维度、`mobGriefing` 和任务边界状态；
- 取消/失败终态之后的导航和世界动作；
- 请求数量、实际破坏数量和实际转移数量。

在正式实验前执行至少一个 mutation/fault-injection 测试，故意破坏每一类不变量，并证明判定器能够报告
对应违规。如果判定器未检出，正式批次不得启动。

## 9. 排除与故障规则

允许排除：

- 协议、候选、实现或场景哈希不匹配；
- 日志损坏且无法从原子检查点恢复；
- Minecraft/Java 进程崩溃、测试机断电等与系统条件无关的基础设施故障；
- 在预注册中列明的外部服务故障；游戏内阶段使用候选回放后原则上不存在该项。

不允许排除：

- 超时、寻路失败、错误拒绝、错误接受或安全违规；
- 与假设方向相反的结果；
- “看起来像异常”但没有预注册基础设施证据的 episode。

所有重跑必须保留原 episode、重跑原因和关联 ID。主要分析遵循预先声明的 intention-to-test 规则。

## 10. 统计分析

### 10.1 主要对比

1. `IBC_PRE` vs `SCHEMA_ONLY`：执行前语义验证；
2. `IBC_MONITOR` vs `IBC_PRE`：运行时检测；
3. `IBC_FULL` vs `IBC_MONITOR`：本地恢复。

### 10.2 分析单位

同一候选、场景模板、布局种子和重复号构成配对单元。主要推断必须对场景模板聚类；机械重复不能作为
完全独立样本。

### 10.3 报告方式

- 报告配对绝对差、相对风险/优势比和 95% 置信区间；
- 小样本优先使用场景级精确置换/符号方法；样本足够时使用带场景随机截距的混合效应模型；
- 三个主要对比使用 Holm 方法控制家族错误率；
- episode 级 McNemar 只作为补充，不作为忽略聚类的主证据；
- 安全事件若为零，报告精确上界或置信区间，不写“证明绝对安全”；
- 缺失值、排除和重跑均单独列出；
- 探索性分析必须明确标注，不与预注册检验混写。

## 11. 工程验收条件

正式 v0.5 冻结前必须全部满足：

- [ ] Schema 验证器与 Java 解码器在生成/边界样本上 100% 一致；
- [ ] 采集任务按请求总量检查背包容量；
- [ ] 重启恢复保留剩余超时预算和完整契约谓词；
- [ ] 恢复总次数存在统一、可测的硬上限；
- [ ] 非本地远端端点必须使用 HTTPS；无效配置必须真正禁用 LLM；
- [ ] 四个架构条件有显式单元测试；
- [ ] 采集、存放、组合、取消、重启和扰动路径有 GameTest/集成测试；
- [ ] 候选回放在不同架构条件中产生完全相同的候选哈希；
- [ ] 安全判定 mutation 测试全部通过；
- [ ] `gradlew test build` 和 Python 分析测试全部通过。

在第一条正式 v0.5 数据产生之前：

1. 工作树必须干净；
2. 创建 Git 提交和不可移动版本标签；
3. 生成模组 JAR、源码 JAR、Prompt、Schema、候选、数据集、场景、分析脚本和环境清单的 SHA-256；
4. 将本文件定稿并发布到 OSF、Zenodo 或学校认可的带时间戳平台；
5. 为代码和数据分别选择明确许可证；
6. 创建 `CITATION.cff`、作者贡献说明和数据字典；
7. 记录 Minecraft、Fabric、Java、操作系统、CPU、内存和模型端点版本；
8. 原始日志只追加不覆盖，派生表必须能从空输出目录重建。

- 不覆盖原协议和原数据；

- 同时报告原预注册分析和修订分析；
- 不把 v0.4 或失败的 v0.5 批次隐藏为“预实验”。
