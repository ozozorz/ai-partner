# AI Partner v0.8 真实游戏验收报告

## 1. 验收范围

本报告记录 v0.8“复杂工作包”在真实 Fabric 开发客户端中的人工验收。测试不是用单元测试或离线模拟冒充游戏运行：客户端实际加载 Minecraft 世界，女仆由服务端逐 tick 寻路、操作方块/熔炉和浮标，结果通过游戏画面、聊天状态、实体 NBT、背包与方块 NBT 交叉核对。

验收环境：

- Minecraft Java Edition 26.1.2；
- Java 25；
- Fabric Loader 0.19.3；
- Fabric API 0.155.2+26.1.2；
- Fabric Loom 1.17.9；
- AI Partner 0.8.0；
- 单人开发世界，客户端与集成服务端同进程。

测试用临时数据包只负责布置可重复场景和发出正式 `/maid` 命令，没有直接修改女仆工作结果；验收结束后已从运行世界移除，不进入源码或发布包。

## 3. 详细验收记录

### 3.1 自然树砍伐

场景同时放置一棵橡树和一组与橡木木板相连的人工原木/树叶结构，女仆背包提供钻石斧，工作模式设为 `lumberjack`。

最终结果：

- 自然树的批准原木计划全部执行完毕；
- 女仆库存得到 `oak_log ×5` 和真实树叶掉落的 `oak_sapling ×2`；
- 钻石斧仍在库存且耐久由原版破坏流程消耗；
- 人工结构断言 `PROTECTED_STRUCTURE_REMAINED` 成立。

首次实机运行发现目标导航错误地指向树冠最高原木，使地面女仆反复寻路。修复后，导航始终锚定树根，实际破坏仍按已批准计划从树冠向根部进行，既保持可达性，也不会在半棵树状态下重新误判。

### 3.2 安全暴露矿石

场景放置两块钻石矿：一块具有水平安全站位，另一块紧邻岩浆。女仆背包提供钻石镐，工作模式设为 `miner`。

最终结果：

- 安全矿石被真实镐破坏并产生 `diamond ×1`；
- 危险目标保持原样，断言 `DANGEROUS_ORE_REMAINED` 成立；
- 女仆没有开挖隧道、破坏围岩或进入岩浆邻接站位。

### 3.3 原版熔炉与跨进程恢复

女仆背包提供 `raw_iron ×4` 和 `coal ×1`，工作区放置空的普通熔炉，工作模式设为 `smelter`。

测试步骤为：女仆租用熔炉并放入批次，确认熔炉点燃后保存退出整个 Minecraft/Java 进程，再重新启动客户端并进入同一世界。开发环境的 Loom 会生成随机离线玩家名，因此只为测试重新绑定了主人 NBT；熔炉、女仆复杂工作阶段和库存均沿用原存档。

最终结果：

- 重启后状态回到 `SMELTER_WAIT`，并继续复验同一原版熔炉；
- 完成后女仆收回 `iron_ingot ×4`；
- 早先库存里的可烧炼原木也在后续循环得到 `charcoal ×2`，符合持续工作会依次处理全部合法输入的设计；
- 没有观察到输入、燃料或产物复制/静默丢失。

### 3.4 通用物流

执行正式命令：

```text
/maid transfer minecraft:amethyst_shard 3 8
```

契约通过验证并完成后，使用方块 NBT 检查目标单箱，得到精确的 `count: 3, id: "minecraft:amethyst_shard"`。任务没有搬走其他物品，状态进入 `COMPLETED`。

### 3.5 真实钓鱼

工作区布置有实体岸线和水面，女仆背包提供钓鱼竿，工作模式设为 `fisher`。

首次实机运行暴露了岸线几何错误：旧逻辑把站位放在与水方块相同的 Y 层；自然岸边的这一格会被水流占据，无法形成合法站位。修复后，`FishingSiteGeometry` 明确区分水面、岸上站位和抛竿水道，并增加几何回归测试。

重启客户端加载修复后，画面确认女仆站在石岸、钓线连接真实浮标且浮标位于水面；`/maid work` 显示 `FISHER_WAIT_BITE`。继续运行多个原版咬钩周期后，女仆库存包含：

```text
fishing_rod ×1
cod ×7
salmon ×2
tripwire_hook ×1
```

这验证了真实抛竿、咬钩、收竿、原版钓鱼战利品实体和跟踪拾取的完整闭环，而不是固定计时后直接生成物品。

### 3.6 成长与状态

多轮工作后的 `/maid status` 输出包含：

```text
Mode: WORKING
job: TRANSFER_ITEM
contract: COMPLETED
work: fisher
growth: Lv.3 / XP 107
affection: 21
```

说明复杂工作奖励、通用任务状态、成长经验和好感度已同步到服务端状态投影。测试结束前执行 `/maid work none`，专属浮标被清理，随后正常保存退出。

## 4. 实机发现并修复的问题

1. 砍树导航目标位于树冠，地面实体无法稳定抵达：改为导航树根、执行已批准列表的树冠到根部顺序。
2. 钓鱼岸边站位与水面同高，真实岸线无法满足：抽取 `FishingSiteGeometry`，站位改为相对水面的上层端点，并为对齐、距离和抛竿水道加入测试。
3. 全局 `mobGriefing` 门禁会误阻止熔炉和钓鱼：规则接口现显式声明是否修改世界，只有砍树、挖矿等规则需要该游戏规则。

## 5. 自动验证与运行日志结论

最终源码另通过完整 Gradle 测试和构建；81 项测试覆盖复杂工作安全策略、状态注册、守恒、成长和几何边界。开发客户端最后一次正常退出，未出现 AI Partner 的异常、Mixin 注入失败或崩溃。

日志中的开发客户端认证/Realms `401` 和客户端资源输出目录缺失提示属于未登录开发环境的预期警告，与模组功能无关。

## 6. 当前边界

- 砍树采用保守启发式；宁可漏掉形态异常的树，也不冒险破坏邻接玩家木结构。
- 挖矿仅处理已暴露的八类原版矿石，不开隧道，也不规划垂直矿道。
- 熔炉只租用普通、空闲且可安全管理的原版熔炉；持续模式会处理库存内后续所有合法可烧炼物。
- 通用物流仍只面向最近的可访问普通单箱，不扩展双箱、木桶、潜影盒或模组容器。
- 钓鱼要求工作区内存在可达岸线和足够的直线水面；不会建造钓鱼台或修改地形。
- 本轮是针对高风险主路径的真实人工验收；大规模自动 GameTest、长时间性能和更多异常注入留到 v0.9 稳定性阶段。
