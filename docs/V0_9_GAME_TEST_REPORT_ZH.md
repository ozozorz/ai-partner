# AI Partner v0.9 真实游戏验收报告

## 1. 验收范围

本报告记录 v0.9“制作与工作补给”在真实 Fabric 开发客户端中的人工验收。测试实际启动 Minecraft、进入单人世界，并由集成服务端逐 tick 驱动女仆的菜单、制作、寻路、方块操作和实体交互；单元测试只作为补充，不替代本报告的游戏内结果。

本轮没有通过自然语言入口发出任务，也没有调用或扩张 LLM。测试用临时数据包仅负责重置场地、放置原版方块/实体和提供确定的初始材料，没有直接生成女仆的制作结果或工作产物；验收完成后已移除。

验收环境：

- Minecraft Java Edition 26.1.2；
- Java 25；
- Fabric Loader 0.19.3；
- Fabric API 0.155.2+26.1.2；
- Fabric Loom 1.17.9；
- AI Partner 0.9.0；
- 单人开发世界，客户端与集成服务端同进程。

## 3. 详细验收记录

### 3.1 固定宽度菜单与具体工作按钮

潜行交互打开女仆菜单后，确认左侧实体/状态、中央 36 格存储与玩家背包、右上 2×2 制作区和右侧控制栏能够在 856×512 的开发窗口内完整显示。生活/日程页可以切换到工作列表；工作列表按三页显示：

1. 关闭、普通作物、甘蔗、瓜类、可可、花草；
2. 除雪、采蜜、剪毛、挤奶、喂主人、喂动物；
3. 插火把、灭火、伐木、暴露矿石、熔炼、钓鱼。

在第三页点击“伐木”后，左侧按钮立即变为 `Work: Lumberjack`，关闭面板后的聊天反馈为 `Persistent work mode set to Lumberjack.`，说明按钮并非纯客户端装饰，而是经过既有菜单白名单和服务端处理器改变权威工作状态。

### 3.2 女仆自身的 2×2 制作区

在菜单的 2×2 输入格放入 1 个橡木原木，结果槽显示并可取出 4 个橡木木板。关闭菜单时未取出的输入会按原版菜单清理语义退回玩家背包或掉落，不把瞬时制作格写入实体持久化数据。

结果由服务端配方管理器匹配普通有形/无形配方，客户端只展示同步结果；测试没有使用客户端直接生成物品的旁路。

### 3.3 空背包徒手伐木

场景清空女仆存储区，移除附近工作台并放置一棵可识别的橡树，然后选择伐木工作。由于背包没有斧头，也没有足以完成工具链的初始材料，物资准备控制器返回规则允许的徒手降级。

约 5 秒后树干已被实际破坏，`/maid inventory` 显示：

```text
minecraft:oak_log x3
```

随后女仆还能利用新取得的木材继续补足制作链。该结果验证“没有斧头”不再令伐木规则永久停摆；徒手路径仍使用真实破坏时间、掉落和拾取逻辑，并采用比持斧更长的工作冷却。

### 3.4 无现成工作台时自动制作木斧

场景移除附近工作台，向女仆背包提供 9 个橡木木板，再开始伐木。观察到的流程为：

```text
个人 2×2 无法制作斧头
  → 用 4 个木板制作工作台
  → 在工作边界内安全放置工作台
  → 导航到工作台
  → 使用 3×3 配方制作木斧
  → 租用木斧并执行伐木
```

最终画面可见新放置的工作台，树木被实际破坏；库存汇总为：

```text
minecraft:wooden_axe x1
minecraft:stick x2
minecraft:oak_log x4
```

这同时覆盖了递归材料规划、工作台制作/放置、3×3 网格边界、工具租用和工作恢复。

### 3.5 个人 2×2 制作剪刀并剪毛

向女仆背包提供 2 个铁锭，附近放置成年绵羊并选择剪毛工作。剪刀配方能够在个人 2×2 网格内完成，因此不搜索或放置工作台。女仆随后对真实 `Shearable` 实体执行剪毛并拾取掉落，库存汇总为：

```text
minecraft:shears x1
minecraft:white_wool x2
```

该场景验证了基础工作规则只声明物资需求，通用补给控制器可以先制作物资，再把控制权交还原工作状态机。

### 3.6 使用已有工作台制作石镐并采煤

场景放置一个可访问的原版工作台、暴露且具有安全水平站位的煤矿石，并向女仆提供 3 个圆石与 2 根木棍。选择“暴露矿石”后，女仆搜索到已有工作台、导航到交互范围，在 3×3 配方中制作石镐，然后由原有安全采矿规则检查工具等级和目标环境并破坏煤矿石。

最终库存包含：

```text
minecraft:stone_pickaxe x1
minecraft:coal x1
```

测试世界中另有一次场景清理产生的 `mutton x1`，与制作或采矿结果无关。石镐与煤的同时存在说明制作后的工具已经进入正式装备租用与采矿链路。

## 4. 实机发现并修复的问题

1. 首版菜单把工作列表直接扩展到 566 像素宽，在小型开发窗口中右侧被裁切：改为固定 382×230 面板，生活/日程与三页工作列表共用右侧栏。
2. 固定侧栏的首轮复测发现分页标题与顶部切换按钮发生文字重叠：移除重复标题，仅保留按钮自身标签和页码，再次启动客户端复验三页均正常。
3. Loom 离线开发客户端重启后随机玩家名变化，旧女仆不属于本次玩家，因此无法打开菜单：测试中重新生成绑定当前玩家的女仆后完成最终 UI 复测。这是开发认证环境特征，不是菜单权限绕过或所有权故障。

## 5. 自动验证与运行日志结论

最终源码通过 `compileJava`、`compileClientJava` 和完整测试集，共 86 项测试、0 失败。测试覆盖服务端制作网格边界、菜单槽位分区、工作按钮白名单、工作注册表以及既有工作/契约回归。

真实客户端最终以退出码 0 正常结束，日志显示集成服务端保存玩家和三个维度后停止，Gradle `runClient` 为 `BUILD SUCCESSFUL`。没有观察到 AI Partner 异常、Mixin 注入失败、崩溃、物品复制或制作原料静默丢失。最终 UI 复测后，测试期间临时设置的“使用物品”键已恢复为 `Right Button`，潜行模式保持 `Hold`。

## 6. 当前边界

- 自动制作只接受服务端配方管理器中的普通有形/无形工作台配方；暂不执行熔炉、切石、锻造台或带特殊动态逻辑的配方。
- 制作规划只消费女仆 35 格普通存储区中的材料，主手租用槽、护甲和副手不作为制作原料，避免误拆当前装备。
- 每 tick 最多提交一个制作步骤，递归深度、候选材料和产量均有上限；找不到守恒且可容纳的方案时不会部分扣料。
- 自动工作台只在工作边界内寻找或安全放置，并受 `mobGriefing` 和放置空间检查约束；不会破坏地形为工作台腾位置。
- 当前只有声明了 `WorkSupplyRequirement` 的工作会主动准备物资。伐木可徒手降级；挖矿等必须工具工作在材料不足时等待并重试，不会绕过工具等级和安全规则。
- 本轮验证了代表性的斧、镐和剪刀供应链；桶、玻璃瓶、锹、火把和钓竿复用同一控制器并有规则/构建覆盖，更多组合与异常注入留给 v0.10 稳定性阶段的 GameTest。
