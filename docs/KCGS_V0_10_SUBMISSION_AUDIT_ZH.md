# AI Partner v0.10 面向《韩国游戏学会论文学报》的投稿审计

> 审计日期：2026-07-22  
> 工程基线：v0.10.0，提交 `87819185fe7ef703b2880ab2362f7ad210b4ca36`  
> 目标：한국게임학회 논문지（Journal of Korea Game Society）研究论文  
> 结论口径：本审计用于研究决策和工程冻结，不构成录用保证，也不替代导师、研究伦理机构或编辑部意见。

## 1. 结论

当前项目是一个功能范围较完整、测试基础较好的 Minecraft/Fabric 研究原型，但还不是可直接投稿的实证论文包。
如果现在提交，最可能落入 **전면수정 후 재투고（全面修改后重新投稿）**，主要原因不是功能缺失，而是：

1. 当前产品代码已发展到 v0.10，论文证据仍来自 v0.4，二者不能视为同一实验对象；
2. v0.4 的成功差异集中在少数场景，场景级统计不支持强因果结论；
3. 变体间没有复用同一份冻结 LLM 候选，语言模型随机性与执行架构效应混杂；
4. 安全测量只覆盖局部方块和部分物品计数，不能支撑“100% 安全”的广义表述；
5. 近期同刊已经出现 LLM 驱动 NPC、LLM 与 FSM、按角色组装原子行为等工作，单纯“用 LLM 控制 NPC”已不构成充分新颖性。

建议把论文主张收窄为：

> 在受控开放世界任务中，指令—行为契约如何通过可拒绝的结构化候选、服务器前置条件、运行时不变量和有界恢复，提高 LLM-NPC 执行的可验证性与故障可诊断性。

论文不应声称已经验证沉浸感、娱乐性、信任、社会临场感，或对全部游戏、任务和语言模型具有普适优势。

## 3. 本次工程与证据审计

### 3.1 当前工程基线

- Fabric/Minecraft Java 模组，Java 25，约 178 个主/客户端 Java 文件、约 2.1 万行主代码；
- 已实现 AI 女仆实体、17 种工作模式、战斗、日程、库存/制作，以及 v2 自然语言驱动；
- LLM 仅产生受类型约束的高层意图，服务器仍负责目标解析、权限、状态机和世界修改；
- 本次整改后共有 144 个 JUnit 测试，完整 `gradlew test` 通过；
- 有 v0.7–v0.10 实机验收记录，但没有 Fabric GameTest 或等价的自动世界集成测试；
- 没有 CI、LICENSE、CITATION.cff、贡献指南和版本化研究制品发布。

工程成熟度可评为“较成熟原型”，但不能用功能数量替代内部效度、统计效度和可复现性。

### 3.2 v0.4 证据复验

本次使用仓库脚本重新生成 v0.4 分析：

- 主实验：162 episodes；
- A2 消融：54 episodes；
- 离线语言评测：72 条指令；
- 5 个 Python 分析测试通过；
- 7 个稳定 CSV/JSONL 输出与仓库冻结输出的 SHA-256 一致。

主要结果应按以下强度解释：

| 结果 | 观察值 | 可支持的解释 |
|---|---:|---|
| RULE_BT 完成 | 54/54 | 规则基线在冻结场景已达天花板 |
| MAID_IBC 完成 | 54/54 | 完整契约机制可完成冻结任务 |
| LLM_SCHEMA 完成 | 48/54 | 仅有结构化候选仍会在扰动下失败 |
| A2 联合消融完成 | 48/54 | 去除监测与恢复会复现相同量级失败 |
| 场景级符号检验 | `p = 0.5` | 不能宣称已有稳健总体优势 |
| DeepSeek 意图分类 | 88.9% | 自然语言意图覆盖有潜力 |
| 规则意图分类 | 47.2% | 规则解析的语言覆盖较窄 |
| DeepSeek 完全匹配 | 38.9% | 槽位精确性仍不足 |
| DeepSeek 半径 F1 | 5.0% | 提示词/默认值协议存在严重偏差 |

6 个 episode 级成功差异实际来自 2 个场景各重复 3 次，不能把 54 个重复 episode 当作 54 个独立场景样本。
因此 v0.4 只能作为机制试点和效应范围参考，不能作为 v0.10 的确认性结果。

### 3.3 证据链的关键断点

#### A. 产品版本与研究版本断裂

v0.10 的对话、控制意图和工作系统已显著扩展，但旧实验运行器仍围绕 v0.4/v1 `JobSpec`。新功能没有经过同一
冻结协议评估，旧数据也不能回填为新功能证据。论文必须明确选择一个新的研究冻结版本。

#### B. 变体间候选不一致

当前批运行器会为不同系统变体分别调用 LLM，而不是复用同一份候选。v0.4 中 15/54 对比较过不同的候选半径。
因此模型采样变化和架构变化被混合，后续实验必须先生成、哈希并冻结候选，再由每个变体重放。

#### C. 消融条件不可归因

现有 A2 同时移除运行时监测和本地恢复，不能区分收益来自前置条件、运行时监测还是恢复策略。新实验至少需要：

- `SCHEMA_ONLY`；
- `IBC_PRE`；
- `IBC_MONITOR`；
- `IBC_FULL`。

#### D. 安全测量范围不足

当前快照主要观察测试区方块和目标物品计数，未形成独立事件账本，无法完整检测区域外修改、错误容器转移、
非目标物品变化、实体伤害、终态后动作、权限/锁/数量越界。旧报告中的“100% safety”只能写成“在已测量约束下
未观察到违规”。

#### E. Schema 与运行时编解码器的一致性

历史 v1 Schema 只强制 `type`，没有按任务类型表达必填参数；它只用于复现 v0.4，因此保留原样并作为 pilot 限制披露。
本轮已修正 v2 的参数为零、基础任务携带参数、空白文本、协议枚举和 Unicode 长度规则，并用 28 组有效/无效
样本同时执行 Draft 2020-12 Schema 与 Java codec。v0.11 新实验只使用 v2，不原地改写 v0.4 的历史协议。

#### F. 本轮已关闭的执行与持久化缺口

- 采集前置条件和每次世界修改前均按完整剩余请求量验证背包容量；
- 当前存档格式持久化并往返测试完整前置条件、目标谓词与不变量；
- 任务快照恢复剩余超时 tick，损坏的超大值会限制到契约最大预算；
- 采集与存放阶段共享契约级恢复预算，耗尽后明确失败。

#### G. 可复现制品仍是本地快照

原快照记录了哈希，但快照时工作树不干净，`artifacts/` 当前未纳入 Git，也没有 DOI/可信时间戳。远程模型名还是
提供商可变别名。后续需要记录调用日期、响应模型标识、完整参数、候选哈希和成本，并发布不可变制品。

## 4. 新颖性定位

同刊已经发表 LLM-NPC 交互、LLM+FSM 设计和隐藏对话等研究；2026 年还出现了按照设计者自然语言意图组合
程序员预制原子行为的角色驱动 NPC 框架。因此论文的新颖性不能停留在“LLM 生成 NPC 行为”或“LLM 与 FSM 结合”。

建议把差异集中在四个可测机制：

1. **拒绝式契约边界**：非法或信息不足候选不会进入世界执行；
2. **运行时不变量**：世界状态变化后持续检查，而不是只在任务开始时验证；
3. **有界局部恢复**：恢复动作、重试预算和终止原因可审计；
4. **配对候选重放**：每个架构变体接收字节一致的候选，使因果比较可解释。

- 冻结并公开预注册，不查看确认性结果后再改主指标。

### P1：正式主实验前必须完成

- 持久化完整契约、剩余超时预算和统一恢复预算（已完成）；
- 增加 Fabric GameTest：重启、卸载区块、多女仆竞争、容器替换和区域外修改；
- 固定随机种子、场景布局、候选、模型元数据和运行环境；
- 先运行不进入主分析的小规模 pilot，并通过安全检测的变异测试；
- 用场景/指令簇作为独立单位制定功效和统计分析方案。

### P2：投稿前必须完成

- 按正式模板压缩为约 10 页韩文或英文论文；
- 图表全部改为英文，参考文献按出现顺序编号并以英文书写；
- 发布代码、冻结数据、分析脚本、环境清单和哈希清单；
- 运行 KCI 相似度检查，完成人工研究伦理与作者贡献声明；
- 等待并核对下一期正式公告、会员状态、费用和上传入口。

## 7. 最终 Go/No-Go 规则

满足下列全部条件才进入投稿排版：

1. 论文描述的提交、Schema、候选库和实验日志哈希完全对应；
2. 预注册中的主分析无事后替换，偏离项逐条披露；
3. 安全账本的已知违规检测变异测试全部通过；
4. 统计单位、置信区间和多重比较处理经过导师/统计审阅；
5. 结论仅覆盖实际测量的任务、扰动、模型和指标；
6. GameTest、单元测试、分析测试和从原始日志重建报告全部通过；
7. 学会当期模板、会员、截止日期、费用和相似度要求均已核验。

任何一项失败均为 No-Go；修复后重新冻结，而不是带着已知证据缺口直接投稿。
